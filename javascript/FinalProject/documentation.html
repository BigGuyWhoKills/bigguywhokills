Here are the highlights of my implementation:
  • My program makes a XMLHttpRequest request to load "gamedata.json" from the same directory, and parses the responseText into a JSON object.
  • That JSON object is passed to a function that walks through the object, moving category data into one array, and expense data into another array.
  • After that, the tables are drawn.
  • A form allows for adding a new category.  When that happens, it is added to the category array and the category table is redrawn.  Additionally, the add new expense form is updated to show the new category.
  • The category table allows the user to edit the budgeted amount for each category.
  • The expense table allows all columns to be edited, aside from the delete-button column.
  • When dates in the expense table are changed, the expenses will re-sort to keep all expenses ordered from oldest at the top to newest at the bottom.
  • The "Save Expenses" button will call a function to save all current game data to local storage.
  • The "Restore Expenses" button will call a function to load previously saved expense data from local storage.
  • This can be tested by changing table data, and saving.  Them refresh the page to load from the JSON file, and use "Restore Expenses" to see the old data you previously saved.
  • A button is provided to delete local storage, and another to show what is currently in local storage.
  • A "Recalculate" button is provided to force an update of the tables, but this no longer needed as my code refreshes the tables after every change.
Here is the rubric for this project, followed by how I have met each requirement:
  1. Game grid that uses CSS and is generated by JavaScript.
    CSS is used to format my table by default.  That formatting is overridden by cellOnClick() in storage.js when a cell's contents are altered.  My buildCategoryTable() and buildExpenseTable() functions in storage.js are the core components that generate the tables in JavaScript.
  2. User input in the form of text, select options, and/or buttons.
    All three of these input options are used on my expenses.html page.  A text input box is provided to add a new category.  A numerical input box in that same fieldset allows the user to designate how much should be budgeted for that category.  A button is used below those two inputs to submit them to my control mechanism.  Finally, a select drop-down list is used on the add-expense fieldset to allow the user to select a category.  That drop-down select is updated every time a new category is added.
  3. User input in the form of mouse clicks.
    Cell in the table (on expenses.html) are clickable, due to onclick handlers that are assigned after the table is built.  Some cells are intentionally not clickable, such as the "Totals" row in the category table.  If a clickable cell is clicked, a prompt window pops up and allows the user to edit the contents.  If the cell value is not changed, or if the prompt window is canceled, and no changes are made to the table.  If a value is changed, the cell background will be changed from black to gray, and the new value will show in the cell.  Immediately after the user clicks "OK", the cell value us updated and the tables are recalculated.  When those tables are recalculated, the background color is reverted back to black.  So the gray background color may not be noticed if the prompt window covers the cell being edited.  Editing the date of an expense will re-sort it within expenseArray[] and on the HTML table.
  4. Dynamic modification of HTML using innerHTML or DOM functions.
    This requirement is satisfied on both the game description page (index.html) and the actual expense tracker (expenses.html).  An easy example to test is the "results" div on expenses.html.  It will show the row number and column number of the cell that is clicked.
  5. Use of XMLHttpRequest to load JSON or XML game data that is displayed on the grid page.
    I load example expense data from 'gamedata.json' in the root directory of this folder or archive.  This can be observed by altering data in that file, and noting the change in the resulting table on the expenses.html page (reload that page to see changes to the file).
  6. Incorporates HTML 5 audio tags, video tags, canvas, or local storage.
    I am using local storage to satisfy this requirement.  This can be observed on the expense tracker (expenses.html) table caption reflecting the userName used to log in to the site (from assignment 5).  Additionally, I have a button that will display rudimentary local storage information in the console log, and another button that will delete local storage created by my project.  I have also implemented the ability to store category and expense information to local storage.  I am currently working on loading that data back into the live display.  This can be tested by altering a few cells, Clicking "Save Expenses", reloading the page to bring in values from gamedata.json, and finally clicking "Restore Expenses" to see the table revert to the changes you previously made.
  7. Game logic code.
    My game logic sums all expense amounts and stores that value in the "Spent amount" column of the category table.  Another function sorts the expenses by their date (in ISO 8601 format).
